### Data Validation Deep Dive
Data validation ensures that all incoming and outgoing data meets the required format, constraints, and business rules. The DEMP backend uses Jakarta Bean Validation (JSR 380) for DTOs, custom validators for complex business logic, and global exception handling for consistent error responses.

**Common Validation Rules:**
- Required fields must not be null or empty (e.g., username, email, event title)
- Email addresses must match a valid email pattern
- Passwords must meet minimum length and complexity requirements
- Dates must be in the future for event creation
- Numeric fields (e.g., ticket quantity, payment amount) must be positive
- Enum values (e.g., ticket status, notification type) must match allowed values

**Custom Business Validation Examples:**
- A user cannot book more tickets than the event's available seats
- Payment amount must match the total ticket price
- Event end date must be after the start date

**Validation Strategies:**
- Use `@NotNull`, `@NotBlank`, `@Email`, `@Min`, `@Max`, `@Pattern`, and other standard annotations on DTO fields
- Implement custom annotations for business-specific rules (e.g., `@ValidEventDate`)
- Centralize validation logic in service layer for cross-field and cross-entity checks
- Return standardized error responses with field, message, and error code

**Example Error Response:**
```json
{
  "timestamp": "2026-05-05T12:00:00Z",
  "status": 400,
  "error": "Bad Request",
  "message": "Validation failed for field 'email'",
  "path": "/api/users/register"
}
```

**Validation Best Practices:**
- Validate all user input at the API boundary
- Fail fast: reject invalid data before processing
- Provide clear, actionable error messages to clients
- Log validation failures for monitoring and auditing

**Advanced Validation Topics:**
- Cross-field validation: Ensure related fields are consistent (e.g., password and confirm password match, event start and end dates are logical).
- Nested object validation: Validate fields within nested DTOs (e.g., address inside user profile).
- Conditional validation: Apply rules only if certain conditions are met (e.g., phone number required if notification type is SMS).
- Uniqueness checks: Ensure values like email or username are unique in the database.
- Bulk/batch validation: Validate lists of objects in a single request (e.g., bulk ticket booking).

**Real-World Validation Scenarios:**
- Prevent duplicate event creation by checking for existing events with the same name and date.
- Validate uploaded file types and sizes for document/media endpoints.
- Ensure payment gateway responses are validated before updating transaction status.

**Sample Java Validation Annotation Usage:**
```java
public class UserRegistrationDTO {
    @NotBlank(message = "Username is required")
    private String username;

    @Email(message = "Invalid email format")
    private String email;

    @Size(min = 8, message = "Password must be at least 8 characters")
    private String password;
}
```

**Custom Validator Example:**
```java
@Target({ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = EventDateValidator.class)
public @interface ValidEventDate {
    String message() default "Event end date must be after start date";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}
```

**Testing Validation:**
- Use unit tests to verify validation logic for DTOs and custom validators.
- Simulate invalid requests in integration tests to ensure proper error handling.

## 12. Advanced Usage & Best Practices

- **API Versioning:** Use URI versioning (e.g., /api/v1/) for backward compatibility. Deprecate old endpoints with clear migration paths.
- **Idempotency:** Ensure POST/PUT endpoints are idempotent where possible, especially for payment and ticketing operations.
- **Rate Limiting:** Apply per-user and per-IP rate limits to prevent abuse. Use HTTP 429 for throttling.
- **Monitoring & Logging:** Integrate with centralized logging (e.g., ELK, Azure Monitor) and set up alerts for error rates and suspicious activity.
- **Testing:** Maintain comprehensive unit, integration, and end-to-end tests. Use test data seeding for consistent test environments.
- **API Gateway:** Route all external traffic through the API gateway for unified security, logging, and throttling.
